# DOCTRINE — RETURN ON COGNITION (ROC)

**Status:** Founder-KPI doctrine. Sits above "founder-hours" as the lens. Derived from deep-research ROC note.
**First written:** 2026-07-03 12:34 PDT

## The law
**The KPI above founder-hours: "how many future work-hours does one hour of thinking delete?"** Ordinary people trade time; managers manage others' time; **system architects delete work.** Optimize the *structure of work getting done*, not the work.

## The single question, for every decision
> **Am I doing work — or am I changing the structure by which that work gets done?**

If you change the structure, the decision pays out *repeatedly*. If you just do the work, it pays once. **Every manual action = assume the system failed**, then ask:
- Can this be a **Rule**?
- Can this be a **Pattern**?
- Can this be an **Automation**?
- Can a **Receipt** prove it?
If yes to any → doing it by hand is **operational debt.**

## The pattern lifecycle (every pattern earns its keep this way)
```
Problem → Observation → Pattern → Rule → Automation → Receipt → Reusable Asset
```
The goal is never to solve a problem once — it is to build an **asset that solves that class of problem repeatedly.**

## Application to the entities
- **SourceA:** every hand-action = a failed system. Convert to Rule/Pattern/Automation/Receipt.
- **Noetfield:** don't produce *decisions* — produce **standardized decision-making** (Decision + Evidence + Policy + Audit-trail). That's what compounds with customer growth.
- **TrustField:** remove friction, don't add process. Each step: "if removed, what's needed to keep trust?" Often a Receipt/Rule/Log replaces several human steps.

## The anti-pattern vs the pattern
| Operator (anti) | Architect (ROC) |
|---|---|
| optimizes today | optimizes the system |
| does more work | deletes repeated work |
| adds people to scale | adds leverage to scale |
| re-solves each problem | turns each repeated problem into a pattern |

## The daily execution order
**Destroy one manual loop per day.** Pick one repeated loop; convert it to a Rule / Template / Receipt-generator / Skill / Automation / Pattern-library entry. The goal today is not to do more — it's that **your future self never has to do that again.**

## Relation to FOUNDER_CANON
ROC is the *why* behind FOUNDER_CANON §1 (zero-founder operational validation) and §3 intent-filter (#1: "does it reduce Sina's future workload — or make Sina the runtime?"). Canon enforces it operationally; ROC is the cognitive lens.
